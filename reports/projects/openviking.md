## 定位

**上下文数据库**——专为 AI Agent 设计，不以传统向量数据库的方式管理上下文，而是用**文件系统范式**组织 Agent 的记忆、资源和技能。

## 核心创新

**1. 文件系统管理范式**

不在 flat 向量库里查 chunk，而是在`viking://` 协议下用目录组织一切：
```
viking://user/memories/        用户偏好/写作风格/编码习惯
viking://agent/memories/       工具使用经验/操作技巧
viking://resources/             连接的数据源
```
支持 `ls`、`find`、`tree`、`grep` 等文件系统命令。

**2. 三级上下文加载 (L0/L1/L2)**

写入时自动把上下文处理成三级：

| 级别 | 内容 | 约 | 对应场景 |
|---|---|---|---|
| L0 (Abstract) | 一句话摘要 | ~100 tokens | 快速相关性判断 |
| L1 (Overview) | 核心信息 | ~2k tokens | 规划阶段决策 |
| L2 (Details) | 完整原始内容 | 全文 | 按需加载 |

类似人类"先扫一眼目录再细读"。

**3. 目录递归检索**

四步走：意图分析 → 向量定位目录 → 目录内二次检索 → 递归子目录 → 聚合。比 flat 向量检索多了"位置感知"。

**4. 可视化检索轨迹**

每次检索的"浏览路径"可追溯，不是黑盒。

**5. 自动会话管理**

会话结束后自动提取长期记忆更新（用户画像 + Agent 经验），Agent 越用越聪明。

## Benchmark

LoCoMo10 提交任务 (1,540 cases)：

| 配置 | Task Completion | Token Cost |
|---|---|---|
| OpenClaw (baseline) | 35.65% | 24.6M |
| OpenClaw + LanceDB | 44.55% | 51.6M |
| **OpenViking** (–memory-core) | **52.08%** | **4.3M** |
| **OpenViking** (+memory-core) | 51.23% | **2.1M** |

比 OpenClaw 基线高 43-49%，比 LanceDB 高 15-17%，token 节省 83-91%。

## 优劣分析

| 维度 | 评价 |
|---|---|
| 范式创新 | 🟢 文件系统范式是差异化最大的方案 |
| Token 效率 | 🟢 L0/L1/L2 三级是实际有效的节省机制 |
| 可调试性 | 🟢 检索轨迹可视化 |
