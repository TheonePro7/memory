## 1. 执行摘要

### 一句话定位

**Agent 记忆的管理层** — 一个独立于任何记忆存储产品的管理平面，让人能够看见、修正、组织和迁移 AI Agent 的记忆，无论底层用的是哪个记忆系统。

### 核心洞察

当前全球 12+ 家公司在做 Agent 记忆存储，但没有一家在做记忆管理。所有人都在帮 Agent "存进去、搜出来"，没有人帮人类管理 Agent 已经记住的内容。结果是：Agent 记错了没人知道，记重复了没人清理，换了记忆系统数据带不走。

我们的判断：AI 的下一阶段不是纯智能体自治，而是**人机协同**。在人机协同时代，人类需要"看得见"Agent 在记什么，"改得了"Agent 记错的内容，"管得住"Agent 记忆的演化。这个需求在 Agent 真正自治之前（5-10 年窗口期）会持续增长。

### 关键数据

| 指标 | 数据 |
|------|------|
| 赛道融资总额（过去 12 个月） | $50M+ |
| 全球竞品数量 | 12+ |
| 国内市场玩家 | 4 家 |
| 直接竞争对手（记忆管理品类） | 0 |
| 当前产品状态 | v0.5 可用（7 个 MCP 工具、86 个测试、Dashboard） |
| 种子轮融资目标 | $1.5M |

### 核心优势

1. **开辟新品类** — 不做第 13 个记忆存储，做第一个记忆管理
2. **聚合锁定** — Mem0 的 Dashboard 只能看 Mem0 的数据，我们能在一个界面里看所有记忆产品。不是功能锁定，是聚合锁定
3. **自有存储是核心而非附赠** — 第一天就是一个完整记忆系统，不依赖任何第三方。竞品全倒闭了我们照常工作
4. **三层托底架构** — 自有备份 + 第三方同步 + 无感切换，永远有数据控制权
5. **切换成本极高** — 用户积累的纠错/标记/整理操作形成强锁定

## 2. 问题与机会

### 2.1 现状：Agent 记忆是黑盒

2024-2026 年，Agent 记忆赛道快速膨胀。全球 12+ 家创业公司涌入，吸引超过 $50M 风险投资。但所有产品都在解决同一个问题：**"如何让 Agent 记住更多"**。

没有人解决一个更基础的问题：**"Agent 记住的东西对吗？"**

**典型场景：**

```
场景 A：Agent 记错了
用户："项目的 deadline 是下周五"
Agent 记成了："deadline = 本周五"
之后所有基于这个记忆的决策都错了
用户没有任何办法发现和修正

场景 B：记忆冲突
Agent 在不同时间记了：
  "用户偏好 React"（2026-01）
  "用户偏好 Vue"（2026-04）
recall 时两条一起返回，Agent 不知道哪条是对的

场景 C：换了记忆系统
用户从 Mem0 迁移到 Zep
历史记忆全部丢失或格式不兼容
用户积累了半年的 Agent 记忆归零
```

**这三个场景有一个共同根因：没有人管理记忆。**

### 2.2 我们的核心判断

```
如果你相信下一个时代是纯 AI 自治 → Agent 自己管记忆就够了
如果你相信下一个时代是人机协同  → 人类必须能管 Agent 的记忆
```

我们的判断是后者。

论据：

1. **技术趋势** — 当前学术界 4 个核心问题（"什么值得记住"、"矛盾如何解决"、"遗忘机制"、"记忆迁移"）全部没有答案。学术界研究"大脑"，工业界做"外挂硬盘"，但**人类仍然需要管理界面**

2. **采用趋势** — Claude Code、Cursor、Windsurf、Codex 等 AI 编程工具的月活开发者已达数十万，这些用户每天产生大量 Agent 记忆，但没有任何人能看到自己 Agent 记住了什么

3. **资本信号** — $50M+ 涌入记忆赛道，说明投资人认定"Agent 需要记忆"。但资本涌入的是"存储基础设施"，不是"管理工具"

4. **确定性需求** — 在企业场景中，"不知道 Agent 记了什么"是不可接受的合规风险。法律、医疗、金融等行业对数据可审计、可修正有刚性需求

### 2.3 市场规模估算

**目标市场：AI Agent 工具与基础设施市场**

| 细分市场 | 当前规模（2026E） | 3 年 CAGR |
|----------|-----------------|-----------|
| AI 编程工具用户 | 50-100 万月活 | 80%+ |
| Agent 平台/框架用户 | 10-20 万月活 | 100%+ |
| AI 应用开发者 | 200-300 万 | 60%+ |

**可寻址市场（TAM）：** $500M+（2028E）

估算依据：
- Mem0 云定价 $19-249/mo，48K ⭐ 开发者
- 假设管理层的价值是存储层的 0.5x-1x（没有存储贵，但用户更愿意为确定性付费）
- 保守估计 $5-10/用户/月 x 500 万开发者（2028）x 10% 渗透率

## 3. 产品定义

### 3.1 产品定位

**记忆管理平面（Memory Management Plane）**— 兼容所有主流记忆存储产品，提供统一的浏览、纠错、组织、迁移能力。

### 3.2 核心功能

#### 第一层：看得见（Memory Browser）

- 统一浏览所有记忆（无论底层用的是哪个记忆系统）
- 来源追溯（这条记忆是谁、在什么上下文、什么时候记录的）
- 冲突标记（存在矛盾记忆时自动标记）
- 时效可视化（旧记忆和新记忆的时间线对比）

#### 第二层：改得了（Memory Editor）

- 编辑/删除单条记忆（记错了立刻改）
- 批量清理（按时间/来源/标签批量操作）
- 优先级标记（告诉 Agent "这条记忆优先使用"）
- 矛盾解决（选择保留哪条、哪条作废）

#### 第三层：管得住（Memory Governance）

- 策略配置：哪些该记、哪些不该记
- 版本回溯：记忆的历史版本可回滚
- 导出/迁移：数据格式标准，跨系统迁移
- 审计日志：谁、什么时间、改了什么

#### 第四层：接得住（Memory Sync Layer）

- 自动同步：Agent recall 时自动注入修正后的数据
- 无感切换：底层记忆系统更换不影响上层使用
- 自有备份：第三方出问题时自动切回备份

### 3.3 用户角色

| 角色 | 需求 | 使用场景 |
|------|------|----------|
| AI 开发者 | 开发 Agent 时需要稳定的记忆调试能力 | 看到 Agent 记住了什么，手动纠错后重新测试 |
| 企业 AI 管理员 | 需要确保 Agent 记忆合规、可审计 | 批量审查 Agent 记忆，清理敏感信息 |
| 重度 AI 用户 | 长期使用 AI 编程工具，积累了海量记忆 | 整理/清理/迁移自己的 Agent 记忆 |

### 3.4 产品形态

| 形态 | 说明 | 定价 |
|------|------|------|
| CLI + MCP Server | 核心引擎，本地运行 | 免费（个人开发者） |
| Dashboard Free | 完整浏览 + 有限编辑（50 次/月） | 免费 |
| Dashboard Pro | 完整 CRUD + 无限编辑 + 批量操作 + 审计日志 | $15/月 |
| Team | 团队协作 + 统一策略 + 合规报告 | $99/月 |
| Enterprise | 自托管 + SSO + SLA | 定制 |

付费转化路径：`免费安装 → 浏览发现记错 → 免费改 50 条 → 不够用 → 升级 Pro`

免费用户改 50 条的过程也是产品在积累价值的过程——改得越多，切换成本越高。

## 4. 技术架构

### 4.1 核心架构：MCP 代理同步层

```
Agent → remember() / recall()  MCP 工具
                ↓
          ┌─────────────┐
          │  记忆同步层    │  ← 核心创新
          │  (core.py)   │
          └──────┬──────┘
                 │
          ┌──────┴──────┐
          ↓              ↓
    ┌──────────┐  ┌──────────┐
    │ 自有存储   │  │ 第三方记忆 │
    │ (ChromaDB │  │ (Mem0 /  │
    │  + SQLite)│  │  Zep /   │
    │          │  │  Letta)  │
    └──────────┘  └──────────┘
          ↑              ↑
          └──────┬──────┘
                 ↓
          ┌─────────────┐
          │  管理界面     │
          │  (Dashboard) │
          └─────────────┘
```

### 4.2 三层保障机制

```
第一层：自有存储（默认启用）
  Agent → remember() → 同步层 → ChromaDB + SQLite ← 管理界面
  功能：完整的记忆系统，不依赖任何第三方

第二层：第三方同步（可选启用）
  Agent → remember() → 同步层 → 自有存储 + Mem0/Zep/...
  功能：兼容历史数据，同步写两份

第三层：无感切换（应急）
  第三方不兼容更新 → 关闭同步，自有存储无缝上位
  用户无感知，Agent 无感知
```

### 4.3 兼容策略

| 存储类型 | 兼容方式 | 优先级 |
|----------|---------|--------|
| ChromaDB（自有默认） | 原生支持 | P0 |
| SQLite（自有默认） | 原生支持 | P0 |
| Qdrant（Mem0 后端） | 适配器 | P1 |
| Markdown（Basic Memory 等） | 文件扫描 | P1 |
| 知识图谱（Zep/Graphiti） | 适配器 | P2 |

### 4.4 技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| 协议 | MCP | 行业标准，Claude Code/Cursor 原生支持 |
| 向量库 | ChromaDB | 离线可用，零依赖，中文友好 |
| 嵌入模型 | fastembed | 离线可用，支持中文 |
| 结构化存储 | SQLite | 零运维，单文件部署 |
| 管理端 | React + Ant Design | 成熟，适合数据管理场景 |
| 安装器 | TypeScript CLI（npx） | 一行命令，无需预装环境 |

**画像 2：AI 团队负责人**
- 管理 3-10 人 AI 开发者团队

3. **从记忆管理反哺记忆产品** — 用户用了管理、需要记忆系统 → 我们附赠基础记忆能力
4. **兼容即获客** — "你在用 Mem0？我们兼容它"比"试试我们的记忆产品"转化率更高

## 6. 竞争格局

### 6.1 直接竞争

**目前为零。** 全球 12+ 家记忆产品中，没有一家把自己定位为"记忆管理工具"。它们全部在做存储基础设施。

### 6.2 间接竞争

| 竞争者 | 为什么不是直接竞争 | 潜在威胁 |
|--------|------------------|---------|
| Mem0 | 做存储 API，不做管理界面 | 如果做管理界面，有 48K ⭐ 的社区优势 |
| Zep | 做时间知识图谱，不做人可管理 | 同左 |
| Letta | 做 Agent 运行时，不做独立管理 | 耦合在运行时中，无法独立管理 |
| Basic Memory | 用文件系统当界面 | 功能简单，不是真正的管理产品 |

### 6.3 我们的结构性优势

**关键洞察：竞品无法拒绝被我们兼容。**

```
Mem0 开源 → 我们可以读 Qdrant 的数据 → 无需 Mem0 的配合
Zep 数据在本地 → 我们可以读取 → 无需 Zep 的配合
Basic Memory 存 Markdown → 我们直接读文件 → 无需 Basic Memory 的配合
```

这意味着：**竞品做什么策略都无法阻止我们管理他们的数据。** 加密？他们为了自己能读，不会加密。改格式？我们跟读。

**唯一的防御方式就是不做本地存储，全部上云。** 但即使如此，如果用户选择自托管版本，数据仍然可以被管理。

### 6.4 差异化总结

| 维度 | 竞品 | 我们 |
|------|------|------|
| 定位 | 记忆存储 | 记忆管理 |
| 用户 | Agent | 人类 + Agent |
| 核心能力 | add/search/delete | browse/review/correct/manage |
| 界面 | API only | Dashboard + API |
| 数据关系 | 只有一个后端 | 兼容所有后端 |
| 确定性 | "存进去了" | "存对了，用到了" |
| 竞品策略 | 抢竞品用户 | 兼容竞品数据 |

## 8. 路线图

### Phase 1：验证付费意愿（当前 → 1 个月）

**目标：** 确认用户愿意为记忆管理付费

- [ ] 访谈 10-20 个目标用户（AI 开发者、团队负责人）
- [ ] 展示当前产品（CLI + Dashboard 只读）
- [ ] 验证核心问题：是否想看 Agent 记了什么、是否想改、是否愿意付钱
- [ ] 确定定价区间

**决策门：** 有人愿意付 → 继续。无人愿意 → 停止或调整方向。

### Phase 2：产品基础（1-3 个月）

**目标：** 可用的最小记忆管理产品

- [ ] 修复 17 个已知 bug（安全漏洞 B01 优先）
- [ ] 补零覆盖代码的测试
- [ ] Dashboard 从只读改为有限编辑（浏览全部 + 50 次/月编辑/删除）
- [ ] 首次启动引导：用户安装后看到"你的 Agent 记住了 N 条信息"
- [ ] 同步层架构：core.py 增加同步调度

**决策门：** 免费用户能在 Dashboard 里浏览和修正 Agent 记忆 → 开源发布。

### Phase 3：兼容层（3-6 个月）

**目标：** 兼容主流记忆产品 + 付费升级路径跑通

- [ ] Mem0（Qdrant）适配器 — 读/写/同步
- [ ] Basic Memory（Markdown）适配器

## 11. 风险与对策

### 11.1 致命风险

**风险 A：上游内置记忆导致第三方记忆产品消亡**

如果 OpenAI/Anthropic 内建记忆能力，Mem0/Zep 等第三方记忆产品可能消亡，我们的兼容策略失去对象。

- **概率：** 中
- **影响：** 致命

**对策：**
1. 自有备份层确保即使第三方消亡，管理产品仍可独立运行
2. 大模型内置记忆≠数据开放，人类仍然需要管理界面
3. 即使 OpenAI 做了记忆，也不会做"兼容其他产品的管理工具"

**风险 B：无人付费**

开发者习惯了免费工具。

- **概率：** 中
- **影响：** 致命

**对策：**
1. Phase 1 先验证付费意愿再投入开发
2. 开源核心 + 付费增值（管理界面 Pro/Team/Enterprise）
3. 企业合规需求是硬付费场景（审计、SSO、自托管）

### 11.2 重要风险

**风险 C：竞品自己做管理界面**

Mem0 / Zep 如果加一个管理页面。

- **概率：** 中
- **影响：** 中
- **对策：** 先发优势 + 他们是存储公司，管理不是基因。更重要的是：他们做管理界面也不会去兼容竞品。

**风险 D：技术执行跟不上生态变化**

记忆产品频繁更新，适配器维护成本高。

- **概率：** 中
- **影响：** 中
- **对策：** 优先兼容存储格式（Qdrant、SQLite、Markdown）而非 API。存储格式的变更频率远低于 API。

**风险 E：市场太早期**

2026 年 Agent 记忆的采用率还不够高，管理需求尚不迫切。

- **概率：** 中
- **影响：** 高
- **对策：** 聚焦 AI 编程工具用户（Claude Code / Cursor）这个最活跃的 Agent 用户群体，他们是记忆管理的第一批刚需用户。

## 12. 融资需求

### 种子轮

| 项目 | 内容 |
|------|------|
| 融资金额 | $1.5M |
| 估值预期 | $5-7M |
| 稀释比例 | 20-25% |
| 资金跑道 | 18 个月 |
| 目标 | PMF 验证 + 200 付费用户 + ARR $100K |

### 投资人价值主张

**这不是又一个 AI 基础设施公司。**

我们不做向量数据库、不做 RAG、不做 Agent 框架、不做第 13 个记忆产品。

**我们是 Agent 时代的 DataDog。** 当所有人在做基础设施，我们做基础设施之上的**管理平面**。无论底层用什么，用户需要一个地方看到 Agent 在干什么、记了什么、记对了没有。

**竞争壁垒：**
1. 品类定义者（先发优势 + 品牌）
2. 用户积累的纠错/标记/整理数据（切换成本）
3. 兼容性适配器网络（越多人用支持越多格式）
4. 人机协同时代的结构性需求（不是周期性的）
